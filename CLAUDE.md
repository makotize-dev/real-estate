**ターゲット合格点**：38点以上（例年の合格ラインは37点前後）

## 鉄則（毎セッション・例外なし）

過去のセッションで繰り返し破ってきた行動を、ここに絶対ルールとして固定する。これは毎セッション必ず読まれる。違反したら、その場で直すだけでなく、原因を `study_log` のその日のログに記録する。**ルールはこの4項目を「書き換えて鋭くする」。日付つきの事件記録をここに追記しない**（事件の経緯は `study_log` に残す）。

- **A. 一次確認を先に。** 条文番号・条文文言・citation・改正の数字を書く前に、e-Gov／一次資料を取得して確認する。説明文を1文字でも書く前に取得する。未確認のものは「未確認」と文中で明示するか、書かない。要約版を原文と思い込まない。語句の「ある／ない」は逐語原文で確認してから断定する。**表・リスト・連続項目を書くときは全要素を逐語で確認してから書く**（一部しか確認していないなら、確認範囲を明示し未確認部分は「未確認」と書く。「いろは順で続くはず」のような推測で残りを埋めない）。**「標準的説明」「判例の整理」「体系書での通説」として書くときは、最初から「（逐語未確認）」とフラグを立てる**（指摘されてから出典を確認するのでは遅い）。
- **B. 評価語を使わない。** 「良い」「いい」「鋭い」「正確」「素晴らしい」「面白い」「見事」等の形容詞でユーザーや発言を褒めない。正誤判定（「正解です」「不正解です」）は省略せず行う。
- **C. 捏造しない。** ユーザーの発言を作り出さない。発言の帰属を推測で決めない。問いを投げたらそこで応答を止める（ユーザーの返答を先取りして書かない）。
- **D. 確認を浅く打ち切らない。** 調査は十分な範囲で行う（例：git 履歴は `--all` で見る）。計画・ルールで既に決まっていること（次に進むか・終了手順の実行など）をユーザーに委ねる確認をしない。自分で判断・実行する。

詳細・背景は `memory/feedback_source_accuracy.md`（A）と `memory/feedback_teaching_style.md`（B〜D）を参照。

## セッション開始時の確認手順（必須）

新しい会話が始まったら、学習を開始する前に以下を必ず実行する。

1. 上の「鉄則」4項目を読み直す
2. `study_log/index.md` を読む → 現在のフェーズ・直近の学習内容・次回予定を把握する
3. `study_log/` 内の直近ログ（最新の日付.md）を読む → 前回の続きから始められるようにする。`git log --all --oneline` でローカル `main` がリモートより遅れていないかも確認する
4. ユーザーの発言意図を確認する（何を学びたいか、どこから始めるか）

**ソクラテス式で進める**：いきなり解説や構造図を出さない。まず質問を投げて考えさせ、回答を受けてから補足・修正・深掘りする。

## 学習ログの運用（毎セッション必須）

Claudeとの学習セッションが終わったら、以下を必ず実行する。

### セッション終了時の手順

**1. study_notes に追記**
その日学んだ条文・トピックを `study_notes/{科目}.md` に追記する（条文番号順を維持して挿入）。
これはセッション終了の必須ステップ。学習したのに study_notes へ未追記のまま終わらない。
**追記の前に draft（追記する内容の全文）をユーザーに提示し、確認を受けてから書き込む**（時間がない・複数科目あるといった理由で draft 提示を省略しない。省略したい場合は省略してよいか先に確認する）。
**他法・関連条文のクロスリファレンスを忘れない**：複数法令にまたがる論点（例：民法177条と不動産登記法60条、建基法43条と78条の建築審査会）を追記したら、**両方のセクションで相互に参照リンクを書く**。一方からしか辿れない構造にしない。

**2. セッションログを作成**
`study_log/YYYY-MM-DD.md` を新規作成（テンプレート：`study_log/_template.md`）
その日のうちに複数回話した場合は同じファイルに追記。

**3. サマリーを更新**
`study_log/index.md` の以下を更新する：

- **週次進捗テーブル**：当該週の「達成トピック」「達成度（最低ライン/ストレッチ達成可否）」を記入する
  - 進捗管理の主軸は**達成内容**。時間は参考メモ（任意）に降格する
  - ユーザーは「設定したゴールを達成するまで学習する」スタイルで、時間ではなく達成内容で進捗を測る
  - 週が変わった場合は次の週の行に移る（前の週はそのまま確定する）
- **W1/W2... 日別計画**：各日の「最低ライン」と「ストレッチ」のトピックを記入する。週末セッションで翌週分を策定する
- **科目別カバレッジ**：「学習済み条文・トピック」と「残り・次の予定」を更新する
- **セッション一覧**：新しいセッションを**先頭**に追加（達成トピックを主軸に記載）
- **次回の学習予定**：次に学ぶ条文・トピックを具体的に記入する

**4. コミット＆プッシュ**
```bash
git add study_log/ study_notes/
git commit -m "学習ログ：YYYY-MM-DD（学習内容の一言メモ）"
git push origin main
```

プッシュ後、スマホから以下のURLでログを確認できる：
`https://makotize-dev.github.io/real-estate/study_log.html`

## 開発ガイドライン

### ブランチ運用
- ひとり学習のためレビュー・PRは不要。**`main` で直接作業する**（作業ブランチは作らない）。
- セッション終了時のログは `main` にコミットし、`git push origin main` で完結させる。
- 注意：Claude Code を worktree で起動すると自動で `claude/` ブランチ上になる。`main` で作業するには、worktree ではなくリポジトリ本体のフォルダ（`C:\dev\real-estate`）で起動する。worktree で起動してしまった場合は、コミット後に `git push origin HEAD:main` で `main` へ反映する。

### コンテンツ作成の方針
- 税理士目線での解説を優先（既存知識との比較・つながりを意識）
- 暗記より理解を重視した説明
- 試験頻出事項には明示的にマークをつける（例：`【頻出】`）
- 表・フローチャートを積極的に使い視覚的に整理する

### HTMLコンテンツのスタイル
- スタンドアロン動作（サーバー不要、ブラウザで開くだけ）
- CSSはファイル内に埋め込み（Tailwind CDN + `<style>`）
- スマホ・PCどちらでも見やすいレスポンシブレイアウト
- PWA対応（`manifest.json` / `sw.js` は共通で全HTMLから参照）
